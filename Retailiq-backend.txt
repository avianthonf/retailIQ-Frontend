# ORACLE DOCUMENT — RetailIQ
### Backend Audit · Python / Flask · Generated 2026-03-20

**Scope:** Code-verified Flask backend findings for frontend integration and architecture review.

## Executive Summary

RetailIQ is a large Flask-based retail operating system backend that combines POS, inventory, analytics, forecasting, supplier procurement, developer platform/OAuth, market intelligence, loyalty/credit, embedded finance, WhatsApp automation, OCR/vision, GST/compliance, and multi-store chain workflows.

The backend is organized as an app-factory Flask service with SQLAlchemy, Redis, Flask-Limiter, Celery, Flask-CORS, Marshmallow validation, and mixed JSON response helpers. The dominant tenancy boundary is `store_id`; almost every operational entity is store-scoped.

There are **154 verified API paths** in the generated OpenAPI artifact, but the codebase also contains several routes that return raw `jsonify(...)` payloads or plain dict-like JSON instead of the unified `format_response(...)` envelope. That inconsistency is the most important frontend integration risk.

## 2) Authentication and Authorization

### JWT merchant/staff auth

- **Mechanism:** Bearer JWT access tokens in `Authorization` header.
- **Access token payload claims:** `sub`, `user_id`, `store_id`, `role`, `chain_group_id`, `chain_role`, `iat`, `exp`, `type`.
- **Refresh token:** opaque token stored in Redis with key `refresh_token:<token>`.
- **Default expiry:** config defaults to 24 hours access token TTL and 30 days refresh token TTL; some tests override these values.
- **Login flow:** `POST /api/v1/auth/login` returns `{access_token, refresh_token, user_id, role, store_id}`.
- **OTP flow:** `POST /api/v1/auth/register` creates user + store and sends OTP; `POST /api/v1/auth/verify-otp` returns tokens.
- **Refresh flow:** `POST /api/v1/auth/refresh` expects `{refresh_token}` in JSON body and rotates the refresh token.
- **Logout flow:** `DELETE /api/v1/auth/logout` is auth-protected and may accept `{refresh_token}` in the body to delete the stored Redis token.

### OAuth developer auth

- **Mechanism:** Bearer OAuth access token for `/api/v2/*` endpoints.
- **Token acquisition:** `POST /oauth/token` supports `client_credentials`, `authorization_code`, and `refresh_token` grants.
- **Token storage:** OAuth tokens live in Redis as opaque strings, not JWTs.
- **Scopes:** enforced in `require_oauth(scopes=[...])`.
- **Usage recording:** `/api/v2` responses are passed through an `after_request` hook that records per-app API usage into the database.

### Role model

Confirmed roles:

- `owner`
- `staff`
- chain role `CHAIN_OWNER` via JWT claim

Important role gates:

- Owner-only mutations exist for store configuration, product creation/deletion, stock audits, loyalty program config, market intelligence, forecasting, GST config, finance KYC/loans/treasury, WhatsApp config, and many analytics endpoints.
- Staff can create transactions but are restricted from owner-only operations.
- Chain owner workflows are separated by `chain_role` in the JWT payload.

### Auth edge cases frontend must handle

- `401` for missing/invalid bearer tokens.
- `403` for role/scope violations.
- `423` account lock on repeated failed login attempts.
- `503` on refresh when Redis is unavailable.
- `mfa_required: true` in the login success envelope when MFA is enabled and code is required.

## 3) Response and Error Contract

### Standard response helper

`format_response(...)` / `standard_json(...)` is the preferred envelope and returns:

```json
{
  "success": true,
  "data": {},
  "error": null,
  "meta": null,
  "timestamp": "2026-03-20T...Z",
  "message": "optional"
}
```

### Global Flask error handlers

Registered handlers exist for:

- `400`
- `401`
- `403`
- `404`
- `405`
- `422`
- `429`
- `500`

These return the same `{success:false,error:{code,message}}` pattern.

### Important divergence

Several routes bypass the unified helper and return raw JSON or partially shaped responses, including but not limited to:

- `app/vision/routes.py`
- `app/finance/routes.py`
- `app/dashboard/routes.py`
- `app/nlp/routes.py`
- `app/api_v2/routes.py`
- `app/ai_v2/routes.py`
- `app/receipts/routes.py`
- `app/payments/routes.py`
- `app/kyc/routes.py`

Frontend code must therefore normalize both the standard envelope and ad hoc objects like `{"message": ...}` or `{"status": ...}`.

## 7) Cross-Cutting Integration Rules

### Pagination

Patterns observed:

- `page` + `page_size` in transactions and some analytics-style endpoints.
- `page` + `limit` in WhatsApp, loyalty, marketplace, and some other routes.
- Some endpoints return `total` and `pages`; others return only `meta`.
- There is no single universal pagination envelope.

### Date and time

- Timestamps are usually UTC ISO 8601 strings.
- Date filters often expect `YYYY-MM-DD` or `YYYY-MM`.
- Some model fields are date-only and some are timestamp fields; frontend should not assume midnight timestamps for date-only fields.
- Some tests intentionally use naive datetimes for SQLite compatibility.

### Numeric precision

- Money and quantities are stored with `Numeric` precision, but transport formats vary between float and string.
- Marketplace app list returns price and rating as strings.
- Frontend should not assume consistent decimal string formatting.

### Caching and revalidation

- Analytics responses are cached in Redis for 60 seconds.
- Forecasting routes read from `forecast_cache` and may return `404` until the batch job runs.
- There is no ETag/Last-Modified contract confirmed.

### CORS

- CORS is applied to `/api/*` only.
- `/oauth/*` is not covered by the current CORS resource rule.
- If the frontend is cross-origin and needs OAuth token exchange from the browser, it may require an additional proxy or CORS adjustment.

### Idempotency

- No idempotency key header is implemented.
- Transaction creation, payment intent creation, loan/finance mutations, PO creation, and webhook queueing should be treated as unsafe to retry automatically.
- Some read-model rebuild tasks are effectively idempotent by design and tests assert that behavior.

### Request IDs / correlation

- No request/correlation ID header is confirmed.

### HTTP method override

- No method override support is confirmed.

### File uploads

Verified upload endpoints and constraints:

- OCR upload: multipart field `invoice_image`, PNG/JPG/JPEG, max 10 MB.
- Receipt digitization: multipart field `receipt_image`, no explicit route-side size guard verified.

## 8) Risks and Frontend Warnings

- **[HIGH] Mixed response envelopes**
  - Why risky: some endpoints use the standard `success/data/error` envelope and others return raw `jsonify` payloads or partial objects.
  - Impact: frontend normalization bugs and inconsistent error handling.
  - Action: create a response adapter that accepts both shapes.
  - Blocker: yes until normalized in the client.

- **[HIGH] Generated OpenAPI is stale relative to code**
  - Why risky: the artifact includes routes such as `/api/v1/health` that are not present in code.
  - Impact: frontend may wire to nonexistent or obsolete endpoints.
  - Action: regenerate API docs from the current codebase before using them as source of truth.
  - Blocker: yes for documentation-driven implementation.

- **[HIGH] OAuth CORS boundary mismatch**
  - Why risky: `/oauth/*` is not inside the `/api/*` CORS rule.
  - Impact: browser-based developer tooling may fail on OAuth token exchange.
  - Action: route OAuth through same-origin or add explicit CORS coverage.
  - Blocker: depends on deployment topology.

- **[HIGH] Real-time auth is not enforced on Socket.IO connect**
  - Why risky: anyone can connect to `/market` namespace unless client-side controls are added later.
  - Impact: unauthorized subscription and data exposure risk.
  - Action: treat websocket data as potentially public until auth is added.
  - Blocker: security-sensitive apps should treat this as a blocker.

- **[HIGH] WhatsApp send endpoints are effectively queue/log stubs**
  - Why risky: UI might show “sent” semantics even though the backend only logs queue state.
  - Impact: misleading delivery UX.
  - Action: show queued/in-progress status and poll message log rather than assuming delivery.
  - Blocker: acceptable only if the UX explicitly says queued.

- **[MEDIUM] Mixed pagination contract**
  - Why risky: `page_size`, `limit`, `pages`, `total`, `meta` all coexist.
  - Impact: list screens need endpoint-specific adapters.
  - Action: create per-resource query hooks.
  - Blocker: acceptable with client abstraction.

- **[MEDIUM] Numeric serialization is inconsistent**
  - Why risky: floats and strings are both used for money/rates depending on route.
  - Impact: charting, formatting, and rounding differences.
  - Action: normalize decimals in a shared formatter and never rely on raw type alone.
  - Blocker: acceptable with careful client code.

- **[MEDIUM] Several background tasks are thin stubs or wrappers**
  - Why risky: frontend may expect eventual state transitions that do not yet exist.
  - Impact: optimistic flows can stall or never resolve.
  - Action: poll the actual read-model/status endpoint and use conservative UX.
  - Blocker: acceptable for non-critical features, but notable.

- **[MEDIUM] Receipt upload temp file path is platform-specific**
  - Why risky: the route writes to `/tmp`, which is Linux-centric.
  - Impact: local/dev behavior may differ on Windows or restricted containers.
  - Action: do not assume receipt upload cleanup or platform independence.
  - Blocker: acceptable for server-side deployment, risky for local testing.

- **[LOW] Public i18n routes are unauthenticated**
  - Why risky: catalog data is openly readable.
  - Impact: minimal unless sensitive translations are introduced later.
  - Action: no special client auth needed.
  - Blocker: not a blocker.

## 9) Frontend Takeaways

1. Use Bearer JWT headers for merchant/staff app screens.
2. Handle refresh via `POST /api/v1/auth/refresh` with refresh token in the body.
3. Treat `/oauth/*` as a separate developer auth system using opaque bearer tokens.
4. Normalize response parsing across standard envelope responses and raw JSON responses.
5. Build per-domain API clients; do not use one rigid response type for the whole app.
6. Poll status endpoints for OCR and print jobs; do not assume synchronous completion.
7. Respect store scope at the UI layer, but still rely on backend authorization for enforcement.
8. Expect several read models (`analytics`, `forecasting`, `market intelligence`) to be cached or derived and therefore temporarily stale.

## 11) Final Assessment

RetailIQ is a feature-rich backend with a strong store-tenancy model, useful read-model caching, and a broad mix of operational and developer-facing capabilities. The backend is production-oriented in many places, but the frontend integration layer must be defensive because the API surface is not yet uniform in response shape, error shape, pagination shape, and async completion semantics.

The safest frontend strategy is:

- one normalization layer for all responses,
- one auth manager for JWT refresh/logout,
- one OAuth client for developer APIs,
- and route-specific hooks for list/poll/cache-heavy workflows.

### 12.1 Verified Frontend Environment Gap

The backend does **not** define browser frontend variables directly; the frontend still needs its own runtime env layer.

- **`VITE_OAUTH_REDIRECT_URI` [FRONTEND MUST SET for authorization-code flows]**
  - Must match the redirect URI the developer app expects.
- **`VITE_PAYMENT_PUBLISHABLE_KEY` [FRONTEND MUST SET only if a provider SDK is added]**
  - Not confirmed in the backend; no publishable key env var is declared here.

### 12.2 Exact Authentication Contracts

#### Merchant / Staff JWT

- **Authorization header**: `Authorization: Bearer <access_token>`
- **Access token payload claims**:
  - `sub`: stringified user ID
  - `user_id`: integer
  - `store_id`: integer or `null`
  - `role`: `owner` or `staff` or `null`
  - `chain_group_id`: string UUID or `null`
  - `chain_role`: `CHAIN_OWNER` or `null`
  - `iat`: issued-at timestamp
  - `exp`: expiration timestamp
  - `type`: `access`
- **Login response** (`POST /api/v1/auth/login`):
  - success body: `access_token`, `refresh_token`, `user_id`, `role`, `store_id`
  - MFA branch: `{mfa_required: true, message: "MFA code required."}` in `data`
  - invalid credentials: `401` / `INVALID_CREDENTIALS` / `"Invalid mobile number or password"`
  - locked account: `423` / `ACCOUNT_LOCKED` / `"Account temporarily locked due to too many failed attempts"`
  - inactive account: `403` / `INACTIVE_ACCOUNT` / `"Account is not verified"`
- **OTP registration flow**:
  - `POST /api/v1/auth/register` accepts `mobile_number`, `password`, `full_name`, `store_name?`, `email`, `role?`
  - `POST /api/v1/auth/verify-otp` accepts `mobile_number`, `otp`
  - `POST /api/v1/auth/resend-otp` accepts `contact` or `mobile_number`, optional `purpose`
  - `resend-otp` success returns `otp_ttl: 120` and `resend_after: 45`
- **MFA flow**:
  - `POST /api/v1/auth/mfa/setup` requires `password`
  - success returns `secret`, `provisioning_uri`, `message`
  - `POST /api/v1/auth/mfa/verify` requires `mfa_code`
  - success returns `{"message": "MFA enabled successfully"}`
- **Refresh flow** (`POST /api/v1/auth/refresh`):
  - body must contain `refresh_token`
  - success returns a rotated `access_token` and `refresh_token`
  - `503` / `REDIS_UNAVAILABLE` / `"Token refresh not available without Redis"` when Redis cannot be reached
  - `401` / `INVALID_TOKEN` / `"Invalid or expired refresh token"` when token missing/expired
- **Logout flow** (`DELETE /api/v1/auth/logout`):
  - best effort; if Redis is down, returns success message anyway
  - may accept `refresh_token` in the JSON body to delete that token key
- **Password reset**:
  - `POST /api/v1/auth/forgot-password` accepts `mobile_number`
  - `POST /api/v1/auth/reset-password` accepts `token`, `new_password`
  - reset tokens live for 900 seconds in Redis

#### OAuth Developer Auth

- **Type**: opaque Bearer tokens stored in Redis; not JWTs.
- **Auth code flow**:
  - `GET /oauth/authorize` returns app name/scopes and asks the user to POST confirm=true.
  - `POST /oauth/authorize` with `confirm=true` redirects to the developer app with `code` and optional `state`.
  - auth codes live for 300 seconds.
- **Token endpoint**: `POST /oauth/token`
  - supported `grant_type` values: `client_credentials`, `authorization_code`, `refresh_token`
  - request may be JSON or form-encoded
  - success token object: `access_token`, `token_type`, `expires_in`, `refresh_token`, `scope`
- **OAuth error shapes**:
  - missing grant type: `{"error": "invalid_request", "error_description": "Missing grant_type"}`
  - invalid client: `{"error": "invalid_client", "error_description": "Invalid client credentials"}`
  - invalid code / refresh token: `{"error": "invalid_grant", "error_description": "..."}`
  - unsupported grant: `{"error": "unsupported_grant_type"}`
- **Scopes confirmed in code**:
  - `read:inventory`
  - `read:sales`
  - `require_oauth(scopes=[...])` rejects missing scopes with `403` / `INSUFFICIENT_SCOPE`

### 12.3 High-Value Endpoint Contract Supplement

#### Store and Configuration

- `GET /api/v1/store/profile`
  - returns `StoreProfileSchema` dump: `store_id`, `owner_user_id`, `store_name`, `store_type`, `city`, `state`, `gst_number`, `currency_symbol`, `working_days`, `opening_time`, `closing_time`, `timezone`
- `PUT /api/v1/store/profile`
  - partial body, same writable fields as the schema
  - owner-only; seeding of default categories occurs when `store_type` is set for the first time
- `GET /api/v1/store/categories`
  - returns `CategorySchema` array: `category_id`, `store_id`, `name`, `color_tag`, `is_active`, `gst_rate`
- `POST /api/v1/store/categories`
  - requires `name`; optional `color_tag`, `is_active`, `gst_rate`
  - 50-category cap per store
- `PUT /api/v1/store/categories/<id>` / `DELETE /api/v1/store/categories/<id>`
  - delete is soft deactivation when allowed; blocked if products are assigned
- `GET /api/v1/store/tax-config`
  - returns `{taxes: [{category_id, name, gst_rate}]}`
- `PUT /api/v1/store/tax-config`
  - accepts `{taxes:[{category_id, gst_rate}]}` and updates matching categories only

#### Inventory

- `GET /api/v1/inventory` returns `ProductSchema` array with pagination meta `{page, page_size, total}`
- query params: `page`, `page_size`, `category_id`, `is_active`, `low_stock`, `slow_moving`
- `POST /api/v1/inventory`
  - `ProductCreateSchema`: `name`, `category_id?`, `sku_code?`, `uom?`, `cost_price`, `selling_price`, `current_stock?`, `reorder_level?`, `supplier_name?`, `barcode?`, `image_url?`, `lead_time_days?`, `hsn_code?`
  - `uom` allowed values: `pieces`, `kg`, `litre`, `pack`
  - validation requires `selling_price >= cost_price`
  - blank SKU is auto-generated
- `PUT /api/v1/inventory/<product_id>`
  - `ProductUpdateSchema` partial body with the same field set plus `is_active`
- `DELETE /api/v1/inventory/<product_id>`
  - soft-deactivates the product (`is_active = false`)
- `POST /api/v1/inventory/<product_id>/stock-update` and `/stock`
  - `StockUpdateSchema`: `quantity_added`, `purchase_price`, `date?`, `supplier_name?`, `update_cost_price?`
  - creates `StockAdjustment`; may update cost price and append `ProductPriceHistory`
- `POST /api/v1/inventory/stock-audit` and `/audit`
  - `StockAuditSchema`: `items` array of `{product_id, actual_qty}`, optional `notes`
  - owner-only; each item becomes `StockAuditItem`

#### Transactions / POS

- `POST /api/v1/transactions`
  - `TransactionCreateSchema`: `transaction_id` UUID, `timestamp`, `payment_mode` (`CASH|UPI|CARD|CREDIT`), `customer_id?`, `notes?`, `line_items[]`
  - line item: `product_id`, `quantity`, `selling_price`, `discount_amount?`
  - returns `{"transaction_id": "..."}` with HTTP `201`
  - emits webhook event `transaction.created`
- `POST /api/v1/transactions/batch`
  - `BatchTransactionCreateSchema`: `transactions[]` (1..500)
  - service code supports partial-success semantics; frontend should not blindly retry
- `GET /api/v1/transactions`
  - pagination via `page` and `page_size`
  - filters: `start_date`, `end_date`, `payment_mode`, `customer_id`, `min_amount`, `max_amount`
  - returns array entries with `transaction_id`, `created_at`, `payment_mode`, `customer_id`, `is_return`
- `GET /api/v1/transactions/<id>`
  - returns full transaction with `line_items`, `original_transaction_id`, `notes`
- `POST /api/v1/transactions/<id>/return`
  - `TransactionReturnSchema`: `items[]` of `{product_id, quantity_returned, reason?}`
  - owner-only; returns `return_transaction_id`
- `GET /api/v1/transactions/summary/daily`
  - query param `date` defaults to today; must be `YYYY-MM-DD`

#### Receipts, Print Jobs, and Barcodes

- `GET /api/v1/receipts/template`
  - returns receipt template fields or default fallback fields if absent
- `PUT /api/v1/receipts/template`
  - accepts `header_text`, `footer_text`, `show_gstin`, `paper_width_mm`
- `POST /api/v1/receipts/print`
  - body: `transaction_id?`, `printer_mac_address?`
  - if `transaction_id` is present, it must be a valid UUID and refer to a transaction in the current store
  - returns `{"job_id": <int>}` with `201`
  - `PrintJob.status` starts at `PENDING`
- `GET /api/v1/receipts/print/<job_id>`
  - returns `job_id`, `store_id`, `transaction_id`, `job_type`, `status`, `created_at`, `completed_at`
- `GET /api/v1/barcodes/lookup?value=...`
  - `value` is required
  - barcode values are validated as alphanumeric plus hyphens, length 4..64

#### Vision / OCR

- `POST /api/v1/vision/ocr/upload`
  - multipart field: `invoice_image`
  - allowed extensions: `png`, `jpg`, `jpeg`
  - max size: `10 MB`
  - creates `OcrJob` with status `QUEUED` and enqueues `process_ocr_job.delay(job_id)`
  - returns `{"job_id": "..."}` with `201`
- `GET /api/v1/vision/ocr/<job_id>`
  - returns `job_id`, `status`, `error_message`, `items[]`
  - item fields: `item_id`, `raw_text`, `matched_product_id`, `product_name`, `confidence`, `quantity`, `unit_price`, `is_confirmed`
- `POST /api/v1/vision/ocr/<job_id>/confirm`
  - body: `confirmed_items[]` with `item_id`, `quantity`, `matched_product_id`, optional `unit_price`
  - only valid when job status is `REVIEW`
  - returns `{"message": "Stock updated successfully"}`
- `POST /api/v1/vision/ocr/<job_id>/dismiss`
  - valid when status is `REVIEW`, `QUEUED`, or `PROCESSING`
  - sets job status to `FAILED` and error_message to `Dismissed by user`
- `POST /api/v1/vision/shelf-scan`
  - body requires `image_url`
  - returns raw analysis payload; shape is defined by the AI helper and should be treated as opaque unless traced in the helper
- `POST /api/v1/vision/receipt`
  - multipart field `receipt_image`
  - saves to `/tmp/<uuid>.jpg` and returns raw digitization result

#### Payments, KYC, and Finance v2

- `GET /api/v1/payments/providers`
  - query param `country_code`, defaults to `IN`
  - returns provider array with `code`, `name`, `type`, `supported_methods`
- `POST /api/v1/payments/intent`
  - requires `transaction_id` and `provider_code`
  - response is the adapter’s raw intent payload
  - stores `PaymentRecord` with `PENDING` status when provider lookup succeeds
- `GET /api/v1/kyc/providers`
  - query param `country_code`, defaults to `IN`
  - returns `code`, `name`, `type`, `id_label`, `required_fields`, `is_mandatory`
- `POST /api/v1/kyc/verify`
  - requires `provider_code`, `id_number`; optional `country_code`
  - returns `{status, details}` where `status` mirrors the provider result
- `GET /api/v1/kyc/status`
  - returns KYC records list with `provider_name`, `status`, `country_code`, `verified_at`
- `/api/v2/finance/*`
  - raw `jsonify(...)` responses are used in many routes; expect shapes like `{"message": ...}`, `{"status": ...}`, and list-only arrays
  - sensitive routes are owner-only (KYC submit, loan apply/disburse, treasury config)

#### OAuth-protected API v2

- `GET /api/v2/inventory?store_id=...`
  - requires `read:inventory`
  - returns `{id, name, sku, stock, price}` items
- `GET /api/v2/sales?store_id=...`
  - requires `read:sales`
  - returns `{id, total, created_at}` items
- Usage of any `/api/v2/*` route triggers the `after_request` usage recorder

#### Developer Platform, Marketplace, Suppliers, Chain, WhatsApp, NLP, Market Intelligence, Events, GST, Loyalty, Credit, Forecasting

- **Developer**: registration returns `id`, `name`, `email`, `message`; app creation returns `client_id`, one-time `client_secret`, `scopes`
- **Marketplace**: search/recommendations/RFQ/orders all return `format_response(...)`; order list wraps `orders`, `total`, `page`, `pages`
- **Suppliers / PO**: supplier profile includes `contact`, `analytics`, `sourced_products`, `recent_purchase_orders`; PO receive uses row locks and updates stock transactionally
- **Chain**: `CHAIN_OWNER` is enforced in a custom decorator, and `group_id` ownership is verified again in the route body
- **WhatsApp**: send routes create log records with status `QUEUED` or `SENT`; the backend does not prove delivery, only queue/log state
- **NLP / AI**: some routes return `{"status": "success", "data": {...}}`, others return `{"response": ...}` or `{"recommendations": ...}` raw; normalize by route family
- **Market intelligence**: summaries, signals, indices, alerts are owner-only and mostly derived from aggregation tables; `acknowledge` flips `acknowledged` to true
- **Events**: `event_type` is limited to `HOLIDAY`, `FESTIVAL`, `PROMOTION`, `SALE_DAY`, `CLOSURE`; dates must be `YYYY-MM-DD`
- **GST**: config uses `GSTConfigUpsertSchema`; summaries can trigger background compilation if a filing does not exist; periods use `YYYY-MM`
- **Loyalty / credit**: reward and repayment flows use nested transactions and row locks; types are `EARN`, `REDEEM`, `EXPIRE`, `ADJUST`
- **Forecasting**: store/SKU routes read from `forecast_cache`; 404 means “not available yet” or “SKU not in forecast tier” rather than a transport failure

### 12.5 Cross-Cutting Implementation Notes

- **Standard envelope**: `{"success": true|false, "data": ..., "error": ..., "meta": ..., "timestamp": ...}`
- **Raw envelope variants**: `{"message": ...}`, `{"status": ...}`, `{"response": ...}`, `{"recommendations": ...}`
- **Pagination patterns**:
  - `page` + `page_size` appears in transactions, customers, inventory, and some list routes
  - `page` + `limit` appears in loyalty, WhatsApp, marketplace, and some other list routes
  - some responses include `pages`; others include only `total` or `meta`
- **Uploads**:
  - OCR invoice upload uses multipart field `invoice_image` and rejects >10MB
  - receipt digitization uses multipart field `receipt_image` and writes to `/tmp`
- **Socket.IO**:
  - namespace `/market`
  - events: `connection_status`, `subscription_status`, `signal_update`, `alert_fired`
  - subscribe/unsubscribe payload is `{topic: "..."}`
- **Caching**:
  - analytics routes are Redis-cached for 60 seconds
  - forecast routes depend on precomputed cache rows; 404 can mean “not ready yet”
- **Idempotency**:
  - no idempotency key support is confirmed; mutations should not be auto-retried

### 12.6 Frontend Architecture Reminder

- React 18 + TypeScript + Vite + React Router + TanStack Query + Axios + Zod + React Hook Form
- Use one response normalizer for standard and raw envelopes
- Use one auth store for JWT refresh/logout and a separate OAuth client state for `/oauth` / `/api/v2`
- Poll print jobs and OCR jobs instead of assuming synchronous completion
- Keep owner/staff/chain-owner route guards explicit in the router layer
